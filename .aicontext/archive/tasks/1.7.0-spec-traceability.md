# Task 1.7.0: Spec readability and traceability

## Created:
April 7, 2026

## Spec:
[spec-1.7.0-framework-improvements.md](../specs/spec-1.7.0-framework-improvements.md)

## Objective:
Make spec Requirements scannable for completion state, link requirement subsections to the tasks that implement them, and add a lifecycle rule that keeps Requirements stable while letting Decisions move out once they stop being defended. Backfill the 1.7.0 spec as a demonstration.

## Requirements:

### Step 3: Add governance rules to process.md
- [x] Add a "Spec Lifecycle" rule to `.aicontext/rules/process.md` under "Task File Management": **Requirements stay forever** (they describe what was built once shipped); **Decisions stay only while they can still be questioned** (once a decision becomes "just how it works", move it out of the spec to a release-notes doc, brief, or delete it). Specs are not changelogs — they are the current contract plus the open arguments
- [x] Add a "Task Requirements vs Spec Requirements" subsection to `.aicontext/rules/process.md` under "Task File Management" — explains the two-layer model, when each is the source of truth, the checkbox-update timing rule (step close vs task close), and how `/finish-task` and `/check-task` use them

### Step 5: Update step-close and task-close verification
- [x] Update `.aicontext/prompts/close-step.md` to walk task and spec requirements after the step is implemented and check off any that this step delivered **100% unambiguously**. Partial or unclear → leave unchecked, do not guess. **Skip-guard**: if the step touched no behavior covered by any task or spec requirement, note "no requirements affected" in the close-step summary and move on. Distinct from the existing "Elevate to Spec" section, which handles *adding* new requirements/decisions surfaced mid-step
- [x] Update `.aicontext/prompts/finish-task.md` so spec alignment: (a) walks task requirements and confirms each is delivered (catches anything close-step left unchecked), (b) walks spec requirements in the linked subsections and checks each that this task fully delivered, (c) **emits a warning** listing any spec requirement that's still unchecked so the user can decide to deliver it now, defer to another task, or revise the requirement. Task reqs are the gate, spec checkboxes are the consequence

## Completion Notes:

- Delivered the spec/task requirements traceability system: checkbox bullets in spec subsections + `*Implemented by:*` footers, two-layer model (spec = system contract, task = work bundle deliverables), 100% rule at step close, hard-block + warning gate at task close with deliver/defer/revise contract
- Spec lifecycle rule was revised mid-task during a conciseness pass: initial framing was "Requirements stay forever, mark obsolete in place" — replaced with "delete when no longer applies; brief and git history preserve rationale". The first version contradicted the conciseness rule and the brief's role; the second is cleaner
- Conciseness pass was a major refactor — ~50% reduction across rules and prompts. Lesson: when adding rules to existing structures, match the shape of the existing rules. Don't invent subsections and prose for content that fits into bullets
- Open observation surfaced during the work but not in scope: subagent verbosity. The `reviewer` agent dumped 400+ lines per review because (a) it doesn't read standards.md (subagents have isolated context), and (b) my reviewer-invocation prompts had `What to check` / `Output format` sections that overrode the agent's terse defaults. Both are real bugs. Captured as follow-up task `1.7.0-context-preservation` (to be created next)
- Backfill of 1.7.0 spec produced 91 checked + 16 unchecked bullets across 14 subsections. The 16 unchecked were pre-walked manually before close to avoid 16 simultaneous warning prompts at finish-task time
- Granularity guidance for task requirements added to process.md: "behavioral, testable; not technical (file paths / function names / line changes)". Test: a requirement should be checkable by someone who can't see the diff
- Follow-up: `1.7.0-context-preservation` — adds the conciseness rule to a place subagents see (CLAUDE.md or agent definitions), creates `agent-setup.md` shared prompt, fixes the reviewer-invocation pattern
- Follow-up (low priority): 5 prompt files reference `spec-{name}.md` instead of `spec-{task-prefix}-{name}.md` — documentation drift, not a behavior bug. Could be cleaned up in `1.7.0-context-preservation` or deferred
