# Task 1.7.0: Plan and commit hygiene

## Created:
April 8, 2026

## Spec:
[spec-1.7.0-framework-improvements.md](../specs/spec-1.7.0-framework-improvements.md)

## Objective:
Direct follow-up to `1.7.0-context-preservation-rules`. Close three bloat sources that the parent task missed: verbose commit bodies (inflate git history forever), automated chores in task plans (redundant with close-step/finish-task), and manual human steps in task plans (unactionable by the agent). All three follow the same "prevent context/history bloat at known sources" pattern as the parent task.

## Requirements:
<!-- Behavioral, testable deliverables — a translation of spec requirements into this work bundle's scope. See process.md "Task Requirements vs Spec Requirements". -->
- [x] **Commit bodies must be concise.** `commit.md` should enforce a concise commit body — "why not what", no diff recap, brevity is the default. Reason: commits land in git history forever, so every extra line pays rent on every future `git log` read
- [x] **Task plans should not include spec or brief updates as steps.** `close-step.md` and `finish-task.md` handle these automatically, so listing them as plan steps is both redundant and pollutes the plan
- [x] **Task plans should not include manual human steps.** No human verification, manual testing, manual QA, user approvals, "check in browser", etc. The plan is for the agent — if the agent can't execute it, it's not a step. *Testing* here means **manual testing only** — automated test runs (via `test-runner`) are agent actions and belong in plans as normal. Human verification belongs in task requirements (as a checkbox gate) or as an interactive question at close time, never as a plan step
- [x] **Task requirements use "should" voice** — state target behavior, not describe current state
- [x] **`/next-step` should chain into `/run-step` after finishing its housekeeping.** `/next-step` keeps its own responsibilities (close the previous step, reflect on whether the plan needs adjusting) and then hands off to `/run-step` to execute the next step — no duplicated execution logic
- [x] **Brief `## Decisions` section renamed to `## Decision Overrides`.** The section is a supersession log, not a store of decisions, and the old name collided with the spec's `## Decisions` section
- [x] **`close-step.md`, `finish-task.md`, `check-task.md` reframed around Decision Overrides semantics.** They currently describe a workflow where brief decisions "graduate to the spec", which contradicts the Brief content boundary rule — non-contradicting decisions go directly to the spec; only supersessions go to the brief
- [x] **`start-feature.md` Step 5 references the renamed section**
